## 🐛 BUG ANALYSIS - CRITICAL ISSUES FOUND

### **BUG #1: Duplicate Database Systems** ⚠️ CRITICAL
**Location:** Everywhere  
**Problem:** 
- 2 database systems running in parallel:
  1. **SQLAlchemy ORM** (app/models) - Modern, migrations
  2. **Legacy db.py** (sqlite3 raw) - Old system
  
**Evidence:**
```python
# File: api.py
from app.models import User  # ← SQLAlchemy ORM
db_manager = Database()      # ← Legacy raw sqlite3

# Registration endpoint uses BOTH:
user_result = auth_service.create_user(username, real_pass)  # SQLAlchemy
db_manager.save_data(features)                                # Legacy
```

**Impact:**
- Data inconsistency risk (2 sources of truth)
- Race conditions possible
- Harder to debug
- Database migrations tidak sync

**Solution:**
- Phase out `db.py` completely
- Migrate semua fungsi ke SQLAlchemy models
- Keep only ONE database access layer

### **BUG #2: Password Hash Stored Twice** ⚠️ SECURITY
**Location:** `api.py` line 191-196  
**Problem:**
```python
# First save: SQLAlchemy (CORRECT)
user_result = auth_service.create_user(username, real_pass)

# Second save: Legacy (WHY???)
else:
    db_manager.save_dev_credentials(username, real_pass, password_hash)
```

**Impact:**
- Same password hash stored in 2 places
- If one gets updated, other stays stale
- Security vulnerability (stale hashes)

**Solution:**
- Remove legacy password storage
- Use only SQLAlchemy User model

### **BUG #3: Session Duplication** ⚠️
**Location:** `api.py` line 475-481  
**Problem:**
```python
# Flask-Login session (CORRECT)
login_result = auth_service.login_user_session(user)

# Legacy session (WHY???)
session['username'] = username
session['login_time'] = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
```

**Impact:**
- 2 session systems (Flask-Login + raw session)
- Can cause logout bugs
- Confusion on which session is authoritative

**Solution:**
- Remove raw session dict usage
- Use only Flask-Login (current_user)

### **BUG #4: Enrollment Count Mismatch** ⚠️
**Location:** `api.py` check_username + register_sample  
**Problem:**
```python
# check_username uses BiometricService count
enrollment_status = biometric_service.get_enrollment_status(username)
enrollment_count = enrollment_status['count']

# But register_sample uses legacy database count
db_manager.save_data(features)  # ← Saves to user_vectors

```

**Impact:**
- If BiometricService reads from different table, count wrong
- Progress bar shows incorrect number
- User confused about enrollment progress

**Solution:**
- Ensure BiometricService reads from same table as save_data writes to
- Add transaction safety

### **BUG #8: Verification Uses Wrong Threshold** ⚠️ SECURITY
**Location:** `api.py` pre_verify_password line 295-302  
**Problem:**
```python
if stored_hash:
    keystroke_threshold = 0.3  # Modern users
else:
    keystroke_threshold = 0.2  # Legacy users (STRICTER!)
```

**Logic Error:**
- Legacy users (no hash) = MORE STRICT threshold (0.2)
- Modern users (with hash) = LESS STRICT threshold (0.3)
- Should be OPPOSITE! Modern = stricter, Legacy = looser

**Impact:**
- Security backwards
- Legacy users harder to login than modern users
- Modern users easier to impersonate

**Solution:**
```python
if stored_hash:
    keystroke_threshold = 0.2  # Modern: strict (has password + keystroke)
else:
    keystroke_threshold = 0.4  # Legacy: loose (only keystroke)
```

### **BUG #9: Rate Limiting Only on Some Endpoints** ⚠️ SECURITY
**Location:** Multiple endpoints  
**Problem:**
```python
@api_bp.route('/register_sample', methods=['POST'])
@limiter.limit("30 per minute")  # ← Has rate limit

@api_bp.route('/check_username', methods=['POST'])
# ← NO RATE LIMIT! Can spam this endpoint
```

**Impact:**
- Attacker can spam check_username to enumerate users
- No protection on username enumeration
- DDoS vulnerability

**Solution:**
- Add rate limits to ALL public endpoints
- Especially check_username (5-10 per minute)

---

### **BUG #10: SQL Injection Risk** ⚠️ CRITICAL SECURITY
**Location:** `db.py` multiple functions  
**Problem:**
```python
# db.py line ~100+
cursor.execute(f"SELECT * FROM users WHERE username='{username}'")
# ↑ F-STRING SQL = INJECTION RISK!
```

**Impact:**
- If username = `' OR '1'='1` → bypass auth
- Can drop tables
- Full database compromise

**Solution:**
- Use parameterized queries EVERYWHERE:
```python
cursor.execute("SELECT * FROM users WHERE username=?", (username,))
```
